## 概述

请针对常见交易流程使用以下订阅和负载检查模式。有关完整结构，请参阅 [Transaction 负载参考](/v2_cn/guides/webhooks-callbacks/wallets-transaction-payload)。

## 充币入账

- 将 `wallets.transaction.succeeded` 作为主要入账信号进行订阅。充币达到 `Completed` 时会触发此事件。
- 订阅 `wallets.transaction.created` 和 `wallets.transaction.updated` 以跟踪交易检测和确认进度，但不要根据这些通知入账。
- 订阅 `wallets.transaction.failed` 以检测被拒绝的充币。例如，交易可能为 `status=Rejected`，且 `sub_status=RejectedByCoboKYT` 或 `RejectedByScreeningApp`。请与发送方对账，不要直接丢弃此类交易。
- 如需独立跟踪 KYT 案件状态，请订阅 `compliance.kyt.screenings.status.updated`。请参阅 [`ComplianceKytScreenings` 事件数据](/v2_cn/guides/webhooks-callbacks/webhook-event-type#compliancekytscreenings-事件数据)。

请使用始终存在的 `transaction_id` 作为交易键，确保余额更新具有幂等性。`cobo_id` 和 `request_id` 均可为 null；填充后，可分别用于交叉核对 Cobo Portal 记录和 API 发起的流程。有关这些字段在交易生命周期中的语义，请参阅 [Transaction 字段](/v2_cn/guides/webhooks-callbacks/wallets-transaction-payload#transaction-字段)。有关使用 `event_id` 进行投递级别去重的信息，请参阅 [Endpoint 范围和对账](/v2_cn/guides/webhooks-callbacks/introduction#endpoint-范围和对账)。如需获取当前状态，请调用 [Get transaction information](/v2/api-references/transactions/get-transaction-information) 或 [List all transactions](/v2/api-references/transactions/list-all-transactions)。

通过 Cobo Portal 和 API 发起的提币采用相同的状态到事件映射，不存在单独的 Cobo Portal 提币事件类型。请同时订阅 `wallets.transaction.created`、`wallets.transaction.updated`、`wallets.transaction.succeeded` 和 `wallets.transaction.failed`，以接收进度以及审批、拒绝或失败的终态结果。

## AutoSweep 交易

当 `cobo_category` 包含 `AutoSweep` 时，该 AutoSweep 交易与其他交易使用相同的 `wallets.transaction.*` 事件路径和 Transaction 负载，不存在单独的 AutoSweep 事件或负载。请参阅 [Cobo 定义的类别](/v2_cn/guides/transactions/manage-transactions#cobo-defined-categories)。

## 通过 Cobo Portal 发起的提币

通过 Cobo Portal 和 API 发起的提币采用[提币跟踪](#提币跟踪)中的相同状态到事件映射。请使用相同的订阅处理程序处理二者。
