`wallets.transaction.created`、`wallets.transaction.updated`、`wallets.transaction.succeeded` 和 `wallets.transaction.failed` 都携带相同的 `Transaction` 数据类型。这些事件的区别在于触发时机，而不是负载结构。触发条件请参阅 [Webhook 事件类型](/v2_cn/guides/webhooks-callbacks/webhook-event-type#交易类事件充币提币合约调用消息签名)。

## 字段参考

| 字段 | 说明 |
|------|------|
| `transaction_id` | 交易 API 操作使用的 UUID，例如 [Get transaction information](/v2/api-references/transactions/get-transaction-information)。调用需要交易路径参数的 API 操作时使用该字段。 |
| `cobo_id` | Cobo 内部交易标识符。它不同于 `transaction_id`，在同时处理 WaaS 1.0 callback 和 WaaS 2.0 webhook 时可用于交易通知去重。 |
| `request_id` | 创建交易时传入的幂等键。对于入账充币，或未提供 `request_id` 的交易，它可以为 `null`。 |
| `wallet_id` | 与交易记录关联的钱包。对于提币，它标识付款钱包；对于充币，它标识收款钱包。 |
| `chain_id` | 链标识符，例如 `ETH`。当一个代币存在于多条链上时，可结合 [List enabled chains](/v2/api-references/wallets/list-enabled-chains) 使用该字段。 |
| `type` | 交易类型，例如 `Deposit` 或 `Withdrawal`。该类型决定 `source` 和 `destination` 中出现的对象结构。 |
| `token_id` | 链维度的代币标识符，例如 `ETH` 或 `ETH_USDT`。请使用该字段进行交易记账、充币入账和对账。 |
| `asset_id` | 不区分链的资产标识符，例如 `USDT`。对于 MPC Wallet、Custodial Wallet 和 Smart Contract Wallet 交易，该字段通常为 `null`；Exchange Wallet 交易可填充该字段。 |
| `source` | 交易来源。其对象结构取决于 `type` 和 `source_type`；完整的 `source` 类型列表请参阅 [Transaction sources and destinations](/v2_cn/guides/transactions/sources-and-destinations)。 |
| `destination` | 交易目标。其对象结构取决于 `type` 和 `destination_type`；完整的 `destination` 类型列表请参阅 [Transaction sources and destinations](/v2_cn/guides/transactions/sources-and-destinations)。 |
| `result` | 仅用于消息签名交易的签名结果，结构为 `{result_type: "Signature", signature}`。它与 `destination` 中的转账金额无关，也不会出现在充币或提币交易中。 |
| `fee` | 交易费用对象。其字段取决于链的费用模型：EIP-1559 使用 gas 字段，UTXO 使用 fee-rate 字段，TRON 使用 fixed fee model。 |
| `confirmed_num` | 当前已获得的区块确认数。当交易处于 `Confirming` 且 `sub_status` 为 `PendingBlockConfirmations` 时，可将其与 `confirming_threshold` 比较。 |
| `confirming_threshold` | 交易离开确认阶段前所需的确认数。状态详情请参阅 [Transaction statuses and sub-statuses](/v2_cn/guides/transactions/status)。 |
| `transaction_hash` | 链上交易哈希。交易广播前为 `null`；链上交易存在后会填充该字段。 |
| `raw_tx_info` | 按链区分的原始交易数据。EVM 交易可包含 `used_nonce` 和 `raw_tx`；UTXO 交易可包含 `selected_utxos` 和 `utxo_changes`。不要将其作为所有链统一的对象结构解析。 |
| `replacement` | RBF 交易的替换详情。当交易被替换时，该对象可包含 `replaced_by_type`、`replaced_by_transaction_id` 和 `replaced_by_transaction_hash` 等字段。生命周期请参阅 [RBF transaction lifecycle](/v2_cn/guides/transactions/rbf-transaction-lifecycle)。 |
| `initiator` | 触发交易的用户、API key 或应用，例如 `API Prod Key #1`。进行操作审计时，请结合 `initiator_type` 阅读。 |
| `initiator_type` | 发起来源渠道，例如 `API`、`Web`、`App` 或 `External`。可用于区分 API 创建的交易与 Portal 或 Portal App 发起的交易。 |
| `block_info` | 交易所在区块的信息，包括 `block_number` 和 `block_timestamp` 等字段。该字段在交易被打包后填充，独立于当前的 `confirmed_num`。 |
| `is_loop` | 表示该转账是否作为 Cobo Loop 转账执行，而不是普通链上广播交易。 |
| `description` | 交易描述。如果未设置描述，可以为 `null`。 |
| `status` | 交易状态。请结合 `sub_status` 使用，不要将 webhook 事件名称解释为完整生命周期状态。状态表请参阅 [Transaction statuses and sub-statuses](/v2_cn/guides/transactions/status)。 |
| `sub_status` | 更具体的状态原因。例如，通过 RBF 被替换的原交易会使用 `ReplacedByNewTransaction` sub-status。 |
| `extra` | 由 JSON 编码字符串组成的列表，用于业务特定的额外数据。每一项都有自己的 `extra_type`，例如 `BabylonBusinessInfo`。 |
| `fueling_info` | Fee Station 创建的自动燃料费交易详情。`main_transaction_id` 会把燃料费交易关联回被补充 gas 的主交易。 |
| `failed_reason` | 审批失败和签名失败时的失败原因。其他失败交易状态不一定填充该字段。 |
| `created_timestamp` | 交易记录创建时间，格式为 Unix 毫秒时间戳。 |
| `updated_timestamp` | 交易记录最后更新时间，格式为 Unix 毫秒时间戳。 |
| `category` | 客户自定义类别标签数组，例如 `Payment`。它不同于由 Cobo 分配的 `cobo_category`。 |
| `cobo_category` | Cobo 分配的类别字符串，用于描述交易的系统类别或用途，例如 `AutoFueling`。它不同于客户自定义的 `category` 标签。高信号取值请参阅 [Cobo-defined categories](/v2_cn/guides/transactions/manage-transactions#cobo-定义的交易类别)。 |

## 充币与提币

以下充币示例使用 `DepositFromAddress` 作为 `source.source_type`，并使用 `DepositToAddress` 作为 `destination.destination_type`。`source` 表示资金来源，`destination.wallet_id` 标识收款钱包。

## EVM、UTXO 与 TRON
