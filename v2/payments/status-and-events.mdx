This document explains all webhook events involved in the payment collection, refund, and settlement processes, as well as status change workflows.

## Webhook events
| Event Name | Trigger Condition | Event Content |
|---------|---------|----------|
| `payment.status.updated` | [**Order mode only**] Payment order status changes. Payment order statuses include:<br></br>- `Pending`: Order created, awaiting payment. No deposit transaction detected yet.<br></br>- `Processing`: First deposit transaction detected at receiving address.<br></br>- `Completed`: Full payment received within order validity period.<br></br>- `Expired`: No completed deposit transactions within order validity period, or order was cancelled.<br></br>- `Underpaid`: Completed deposit transactions exist within order validity period, but received amount is less than required amount. | [PaymentOrder](/v2/payments/status-and-events#paymentorder) |
| `payment.address.updated` | [**Top-up mode only**] Top-up address update | [PaymentAddressUpdate](/v2/payments/status-and-events#paymentaddressupdate) |
| `payment.transaction.created` | New deposit transaction detected. In order mode, this event is triggered even if order has reached final status (`Underpaid`, `Expired` or `Completed`). | [PaymentTransaction](/v2/payments/status-and-events#paymenttransaction) |
| `payment.transaction.late` | [**Order mode only**] Deposit transaction passed compliance screening but order is already in final status (`Underpaid`, `Expired` or `Completed`). | [PaymentTransaction](/v2/payments/status-and-events#paymenttransaction) |
| `payment.transaction.held` | The deposit transaction is being screened for compliance. In order mode, this event is triggered even if order has reached final status (`Underpaid`, `Expired` or `Completed`). | [PaymentTransaction](/v2/payments/status-and-events#paymenttransaction) |
| `payment.transaction.completed` | Deposit transaction passed compliance screening, funds successfully credited and included in received amount. | [PaymentTransaction](/v2/payments/status-and-events#paymenttransaction) |
| `payment.refund.status.updated` | [**Order mode only**] Refund order status changes. Refund order statuses include:<br></br>- `Pending`: Refund order created but transaction not initiated yet.<br></br>- `Processing`: At least one refund transaction in progress.<br></br>- `Completed`: All refund transactions completed.<br></br>- `PartiallyCompleted`: Some refund transactions completed, some failed.<br></br>- `Failed`: All refund transactions failed. | [PaymentRefund](/v2/payments/status-and-events#paymentrefund) |
| `payment.settlement.status.updated` | Settlement request status changes. Settlement request statuses include:<br></br>- `Pending`: Settlement request created, awaiting processing.<br></br>- `Processing`: Settlement request processing, at least one settlement in progress.<br></br>- `Completed`: All settlements completed.<br></br>- `PartiallyCompleted`: Some settlements completed, some failed.<br></br>- `Failed`: All settlements failed. | [PaymentSettlement](/v2/payments/status-and-events#paymentsettlement) |


## Event Details

### PaymentOrder
Refer to response parameters in [Get pay-in order information](/v2/api-references/payment/get-pay-in-order-information).

### PaymentAddressUpdate

This data structure contains the following fields:

- `custom_payer_id`: A unique identifier assigned by the developer to track and identify payers in their system
- `payer_id`: A unique identifier assigned by Cobo to track and identify payers
- `chain`: Chain ID, e.g. "ETH"
- `previous_address`: Previous top-up address used by the payer
- `updated_address`: New top-up address assigned to the payer

### PaymentTransaction

This data structure contains all fields from the [Get transaction information](/v2/api-references/transactions/get-transaction-information) response parameters, plus the following fields:

  - `acquiring_type`: Acquiring type. `Order` indicates order mode, `TopUp` indicates top-up mode.
  - `order_id`: Pay-in order ID.
  - `psp_order_code`: A unique identifier in the developer's system to identify this order.
  - `payer_id`: A unique identifier assigned by Cobo to track and identify payers.
  - `custom_payer_id`: A unique identifier assigned by the developer to track and identify payers in their system.

### PaymentRefund
Refer to response parameters in [Get refund order information](/v2/api-references/payment/get-refund-order-information)

### PaymentSettlement
Refer to response parameters in [Get settlement request information](/v2/api-references/payment/get-settlement-request-information)
