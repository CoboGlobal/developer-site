## Triggers

| Event type | When it occurs |
| --- | --- |
| `wallets.transaction.created` | A transaction record has been created. For an inbound transaction such as a deposit, this follows on-chain detection. For an outbound transaction such as a withdrawal, this follows Cobo creating the record from the request. This event does not indicate that the transaction was successful. |
| `wallets.transaction.updated` | There are changes to a transaction's status or confirmation numbers. |
| `wallets.transaction.succeeded` | A transaction has been successfully completed. |
| `wallets.transaction.failed` | A transaction failed. |

These events apply to every transaction type, including deposits, withdrawals, contract calls, message signing, staking, and unstaking.

### State-to-event mapping

| Transaction status | Additional terminal event |
| --- | --- |
| `Completed` | `wallets.transaction.succeeded` |
| `Failed`, `Rejected`, or `Canceled` | `wallets.transaction.failed` |

The terminal event is attempted in addition to `wallets.transaction.updated`. See [Event overlap for a single transaction](/v2/guides/webhooks-callbacks/webhook-event-type#event-overlap-for-a-single-transaction) for how the event types overlap.

## Webhook envelope

Every delivery contains the following envelope. For these events, `data.data_type` is always `Transaction`.

| Field | Description |
| --- | --- |
| `event_id` | UUID of this delivery record. Use it to deduplicate retries of the same event. |
| `url` | Endpoint URL to which the event was sent. |
| `created_timestamp` | Time the event was triggered, as a Unix timestamp in milliseconds. This value does not change across retries. |
| `type` | Webhook event type. |
| `data` | Object containing the `data_type` discriminator and transaction payload in `data`. |
| `status` | Delivery status for this endpoint: `Success`, `Retrying`, or `Failed`. This is separate from the transaction's business status. |
| `next_retry_timestamp` | Next retry time as a Unix timestamp in milliseconds. Present only when `status` is `Retrying`. |
| `retries_left` | Number of delivery retries remaining. Present only when `status` is `Retrying`. |

### `raw_tx_info`

| Field | Description |
| --- | --- |
| `used_nonce` | Nonce used by the transaction. |
| `raw_tx` | Serialized signed transaction. |
| `unsigned_raw_tx` | Serialized unsigned transaction. |
| `selected_utxos` | UTXO-chain inputs consumed by the transaction. |
| `utxo_changes` | UTXO-chain change outputs created by the transaction. |

### `replacement`

| Field | Description |
| --- | --- |
| `replaced_by_type` | Replacement operation applied to the original transaction. |
| `replaced_by_transaction_id` | Transaction ID of the new transaction that superseded the original. |
| `replaced_by_transaction_hash` | Transaction hash of the new transaction that superseded the original. |
| `replaced_type` | Replacement operation that created the new transaction. |
| `replaced_transaction_id` | Transaction ID of the original transaction. |
| `replaced_transaction_hash` | Transaction hash of the original transaction. |

See [RBF transaction lifecycle](/v2/guides/transactions/rbf-transaction-lifecycle) for the replacement flow and [Transaction statuses and sub-statuses](/v2/guides/transactions/status) for `ReplacedByNewTransaction`.

### `timeline`

Each `timeline` entry contains:

| Field | Type | Description |
| --- | --- | --- |
| `status` | string | Transaction status represented by this lifecycle stage. |
| `finished` | boolean | Whether the transaction has completed this lifecycle stage. |
| `finished_timestamp` | integer, nullable | Time the stage completed, as a Unix timestamp in milliseconds. It is `null` while the stage is unfinished. |

## Field semantics

The `data.data` object contains the transaction fields above. For the distinction between `token_id` and `asset_id`, see [Transaction fields](#transaction-fields).

For the lifecycle-dependent availability of `cobo_id` and `request_id`, see their entries in [Transaction fields](#transaction-fields). `result` appears only for a completed `MessageSign` transaction. Source and destination shapes vary by transaction type; see [Transaction sources and destinations](/v2/guides/transactions/sources-and-destinations).

Replacement fields use the same Transaction shape and link the original and replacement records. Similarly, an AutoSweep transaction, identified when `cobo_category` contains `AutoSweep`, uses the same Transaction shape as any other withdrawal.
